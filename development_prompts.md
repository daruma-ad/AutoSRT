# AutoSRT 開発・運用プロンプト＆ガードレール仕様集

このドキュメントは、本システム（AutoSRT）を構築する際に使用した「開発用指示プロンプト」や、内部でAI（Gemini API）に指示を出すための「システムプロンプト」に加え、**AIの出力崩壊を防ぐために実装した「究極のガードレール」の設計指示**をまとめたものです。

これらを参考にすることで、類似のLLM連携アプリケーション開発や、AIへの的確な指示出し（プロンプトエンジニアリング）、そして**「AIの罠」を回避する堅牢なシステム設計**のノウハウを再利用することができます。

---

## 1. アプリケーション開発指示プロンプト（初期メタプロンプト）

システムの基礎を作り上げる際、AIエンジニアに対して最初に与えた包括的な指示内容です。要件定義、技術スタック、アーキテクチャを明確に規定しています。

```markdown
# 指示
あなたは優秀なフルスタックエンジニアであり、動画・音声処理およびLLMを活用したアプリケーション開発の専門家です。
以下の要件に従って、「動画編集の前処理（無音カットと字幕校正）を自動化するWebアプリケーション」を構築するためのコード一式を生成してください。

# プロジェクト概要
Premiere Pro等での編集作業を効率化するため、ブラウザ上で音声解析による「無音カット（FCP XML生成）」と、最新のLLM（Gemini 2.5 Flash）を用いた「SRT字幕の文脈に沿った自動校正」を行えるアプリケーションを開発します。

# 技術スタック
- フロントエンド・バックエンド: Python + Streamlit
- 音声処理: ffmpeg-python, numpy, scipy
- XML処理: lxml (Premiere Pro用 FCP XML)
- 字幕処理: pysrt
- LLM連携: google-generativeai

# 実装フェーズと要件
## フェーズ1: インテリジェント無音カット機能
1. 音声ファイルからRMS音量を計算し、設定した閾値以下の区間を「無音」と判定する。
2. 音量閾値、最小無音時間、パディングをスライダーで調整可能とする。
3. アップロードされたFCP XMLファイルを読み込み、有音区間に合わせて`<clipitem>`を分割・再構築する。
   ※NTSC（29.97fps）のタイムベース差異による「ズレ」を完全に防ぐこと。

## フェーズ2: コンテキスト考慮型 SRT 字幕校正とスマートフォーマット
1. SRTファイルを受け取り、指定したチャンクサイズごとにGemini APIに送信する。
2. ユーザー指定の「動画の文脈」「専門用語リスト」を厳格に反映させること。
3. 【重要】AIへ送信する前に、以下の「スマートフォーマッティング」を実施すること。
   - テロップが短すぎるエントリ（例: 'あのー'等）を自動的に前後のエントリに結合する。
   - 画面の専有を防ぐため、1行あたりの文字数を計算し、最大2行までに強制改行する。
4. 【絶対要件】元のタイムスタンプと同期を絶対に破壊させないこと。
```

## 2. SRT字幕校正用システムプロンプト（API内部実装）

`srt_processor.py` 内部で、Gemini にSRTデータを渡し、テキスト校正を行わせるためのシステムプロンプトです。**「フォーマットを破壊させないための制約」**を強く組み込んでいます。

```markdown
あなたはプロフェッショナルな映像翻訳・字幕校正者です。
以下に提供されるSRT字幕データのテキスト部分のみを、文脈に合わせて自然な日本語に校正してください。

【厳守事項】
1. **フォーマット維持**: 
   - エントリ番号（1, 2, 3...）
   - タイムスタンプ（00:00:00,000 --> 00:00:00,000）
   上記は元のデータを「完全にそのまま」出力してください。絶対に改変や削除をしないでください。
2. **テキストのみ校正**:
   - タイムスタンプの下にあるテキスト（字幕内容）のみを自然な日本語に修正してください。
3. **無駄な出力禁止**:
   - 余分な装飾やMarkdownによるコードブロックは一切出力しないでください。
```

## 3. 【最重要】LLMの暴走を防ぐガードレール実装プロンプト（デバッグ指示）

LLMはプロンプトでどれだけ「絶対に変えないで」と指示しても、**必ずどこかでミスを犯します**。以下のプロンプトは、実機テストで発見された致命的なバグ（タイムコードの画面への流出や、音声ズレ）をプログラム側で強制的にブロックするために行った「改修指示（ガードレール設計）」の履歴です。

### 3-1. AIの「改行忘れ」を自動修復する正規表現ガードレール
```markdown
# 課題
Geminiが以下のように出力をミスし、前のエントリのテキストと、次のインデックス番号を改行なしで繋げて返してくることがあります。
`...育児休業給付、281\n00:09:53,526 -->`
これによって正規表現のパーサーが限界を迎え、タイムコードが字幕本文として画面に流出するバグが起きています。

# 指示
パースの直前に、テキストを自動修復する正規表現を挟んでください。
条件：「数字と改行文字以外の文字（例：日本語の句点など）」の直後に、「数字＋タイムコードのブロック」が来た場合、間に強制的に改行を2つ挿入するリプレイス処理を実装してください。
```

### 3-2. AIの「インデックス破壊」からタイムコードを守るオフセット照合
```markdown
# 課題
AIが短すぎるエントリを勝手に削除した上で、返すインデックス番号の起点を「1」から強制リセットするという二重のミスを犯すことがあります。位置ベースでタイムコードを復元しようとすると、数秒〜十数秒単位の激しい音声ズレが起きます。

# 指示
タイムスタンプの復元ロジックを、完全に不変な「時間情報」ベースのオフセット計算に改修してください。
1. AIが返した各エントリの `start` の先頭8文字（HH:MM:SS）を使い、原文のチャンクとマッピングする。
2. その差分から本当の「オフセット（ズレ幅）」を計算し、AIが壊したインデックス番号を逆算して強制上書きする。
3. 万が一、照合が完全に不能になった（数が合わずオフセットも特定できない）場合は、絶対に音声ズレを起こさないよう、そのチャンクだけはAIの校正を破棄し「原文（未校正）」をそのまま返すフォールバックを実装すること。
```

## 4. UX改修用プロンプト事例（カットプリセット追加時）

すでにシステムが稼働している状態から、「使い勝手（UX）の向上」を指示する際の効果的なプロンプトの例です。

```markdown
# 課題
動画編集の初心者にとって、「最小無音時間: 167ms」「パディング: 67ms」というミリ秒指定の値は直感的にわかりにくい状態です。

# 指示
StreamlitのUIを改修し、カットのきつさを抽象的に選べる「プリセットボタン」を追加してください。

# 要件
1. 以下の3段階のスタイルを表現できるボタンを配置する。
   - ⚡ ジェットカット（テンポ重視）
   - 🎙️ スタンダード（自然な間を残す）
   - 🎬 シネマティック（余韻を持たせる動画向け）
2. プリセットを選択すると自動的にスライダーに推奨値がセットされること。
3. スライダーを動かした際、現在のミリ秒（ms）が「29.97fps」等において何フレームに相当するか、リアルタイム換算して表示すること。
```
