# Antigravity×Premiere Pro：動画編集の「最もツラい作業」を消し去るAI自動カット＆テロップ校正ツール

動画編集の案件をこなす中で、**一番時間がかかって、一番心が折れる作業**は何でしょうか？

エフェクトの作り込み？ 色調補正？ 音の調整？
いいえ、違いますよね。

**「ひたすら無音を削るジェットカット」と「誤字脱字だらけのテロップ修正」です。**

もちろん、今のAdobe Premiere Proには優秀な自動文字起こし機能や、無音を自動検出する機能も備わっています。「じゃあそれで十分じゃないか」と思うかもしれません。
しかし、現場でゴリゴリ編集している人なら分かるはずです。

*   **「Premiereの自動カットは、微妙な『間』の調整が直感的にやりづらい…」**
*   **「文字起こしは便利だけど、同音異義語の誤変換が多すぎる。結局1文字ずつ目視で直して、文節の改行位置を整えるのに膨大な時間が溶ける…」**

「それならいっそ、最先端のAIエージェントとコードベースの動画生成（Remotion等）を組み合わせて、全て全自動で作ってしまえばいいのでは？」
実際、私もそれを試しました。しかし結論から言うと、**AIを使っても最初から100%の完成度に仕上げるのは、今のところまだ難しい**のが現実でした。クライアントの案件ごとに求められるテロップの細かい表現、演出方法、絶妙なタイミングの微調整において、やはりPremiere Proというソフトは**神がかったUIを持つ超優秀なツール**なのです。

しかし、AIの力を使えば**「一瞬で80%の完成度まで持っていくこと」**は十分に可能です。そこからの最後の仕上げ（20%のクリエイティブな微調整）は、やはりPremiere Proを使うのが一番合理的なアプローチだと確信しました。ゼロからコードで表現をチューニングしていくよりも、80%完成したタイムライン上でマウスを握り、Premiere独自の直感的な操作でクオリティを限界まで引き上げるほうが圧倒的に効率的です。

つまり、私たちが本当に求めているのは「編集ソフトを乗り換えること」や「完全自動化の夢を追うこと」ではなく、**「一番面倒な『無音カット』と『テロップ修正』という泥臭い作業をAIに丸投げして素早く80%の土台を作り、残りのリソースをPremiere Proでのクリエイティブな仕上げに全集中すること」**だったのです。

そして今、最新のAIモデル（Gemini 2.5 Flash等）の精度向上により、それがついに可能になりました。

## 【無料公開部分】AutoSRTでできること

このアプリは、あなたのPremiere Proでの編集前処理を「数クリック」で終わらせます。

### 1. Premiere Proに直接読める「AI自動カット（FCP XML生成）」

音声ファイル（MP3/WAV）と、書き出したFCP XMLをツールに読み込ませるだけで、AIが波形を解析して不要な無音部分を自動でカット。Premiere Proでそのまま読み込める新しいXMLを吐き出してくれます。

さらに、動画のテイストに合わせて**3つのカットスタイル**をワンクリックで選べるようにしました。

*   **⚡ ジェットカット**: テンポ重視。無音をほぼ全て除去し、テキパキした印象に（YouTube・解説動画向け）。
*   **🎙️ スタンダード**: 自然な「間」を少し残すバランス型（インタビュー・1人語り向け）。
*   **🎬 シネマティック**: 余韻や沈黙を大切にする控えめなカット（ドキュメンタリー・感動系向け）。

「最小無音時間」や「パディング（余白）」といった細かなパラメータもスライダーで直感的に微調整可能。しかも「29.97fpsなら何フレームか」がリアルタイム表示されるため、Premiereのタイムライン感覚のまま数値を設定できます。

### 2. Gemini 2.5 Flashによる「超・文脈考慮型」テロップ校正

ここが既存ツールとの決定的な違いです。
元のSRTファイルには、「同音異義語のミス」や「専門用語の誤変換」、そして「短すぎる細切れの字幕」がつきものです。

AutoSRTでは、Googleの最新・超高速AIモデル「Gemini 2.5 Flash」を活用。「動画のジャンル」や「専門用語リスト」を入力しておくことで、AIが動画の**文脈を理解**し、**タイムスタンプ（字幕の表示タイミング）を一切壊さずにテキストのみを完璧に校正**してくれます。

さらに、テレビ番組のような「プロのテロップ」にするための**インテリジェント・フォーマッティング機能**を搭載：
*   **文字数ベースの自動改行**: 1行あたりの文字数を指定するだけで、意味の通る位置で自然に改行。最大2行までに強制制限し、画面を文字で埋め尽くしません。
*   **短すぎる字幕の自動結合**: 「あのー」「それで」といった細切れの字幕を、前後の字幕と論理的に結合し、カチッとした読みやすい文章にまとめ上げます。

### 3. 直感的なUIと「デスクトップ保存」機能

PythonのWebフレームワーク「Streamlit」を採用し、ブラウザ上で直感的に操作できる美しいUIを構築しました。
処理が終わったら、「📥 ダウンロード」ボタンだけでなく**「🖥️ デスクトップに保存」**ボタンも用意。Macのデスクトップに直接ファイルが保存されるので、ダウンロードフォルダをごちゃごちゃ探すストレスもありません。そのままPremiereにドラッグ＆ドロップするだけです。

### 1. FCP XML再構築における「NTSC累積ドリフト」の罠

Premiere Proの無音カットをする際、一番手強いのが「フレームレートの計算」です。最初のテストでは「動画の後半になるにつれて、音と映像がどんどんズレていく（10分で0.6秒のズレ）」という致命的な問題が発生しました。

**原因は「29.97fps（NTSC）」の処理**でした。FCP XMLに記載されている `timebase` は `30` ですが、NTSC動画の場合、実際のフレームレートは正確には `30 × 1000 / 1001 = 29.9700299...` fpsとなります。公称値の「30」のまま掛け算していくと、実際のフレームレートとのごく僅かな差が蓄積してしまいます。

これを解決するため、NTSCフラグを判定し、厳密なFPS計算を導入しました。さらに、Premiere特有の絶対時間単位である `pproTicks` （1秒＝254,016,000,000）を、フレーム境界と完全に同期するよう計算し直すアルゴリズムを開発しました。

```python
def get_actual_fps(timebase: int, ntsc: bool) -> float:
    # NTSCの場合は 1000/1001 を掛けて正確なFPSを算出
    return timebase * 1000.0 / 1001.0 if ntsc else float(timebase)

def sec_to_timebase_frames(seconds: float, timebase: int, ntsc: bool) -> int:
    actual_fps = get_actual_fps(timebase, ntsc)
    # 浮動小数点の誤差を排除するため確実に四捨五入して整数ピクセル（フレーム）を特定
    return int(round(seconds * actual_fps))

# 1フレームがズレないよう、再度「実際のFPS」から秒数を逆算してから pproTicks に変換
frame_aligned_in_sec = in_frame / actual_fps
xml_clip_in = str(int(round(frame_aligned_in_sec * 254016000000)))
```

この「フレームから秒に一旦戻し、それを `pproTicks` に変換する」という設計により、1時間の長尺動画でも**1フレームたりともズレない**完璧なXML出力を実現しています。


### 2. LLMによるSRT字幕校正に潜む「崩壊」と究極のガードレール

SRTファイルをLLM（Gemini API等）に食わせる際、最大の難関は**「AIがタイムスタンプを勝手に変えたり、書き出す数を減らしたりして、字幕ファイルが致命的に壊れる」**ことです。
プロンプトで「フォーマットを絶対に変えるな」と指示しても、LLMは平気で裏切ります。このツールでは、これらを完全に無効化するため以下の**「多重ガードレール」**を構築しました。

#### ガードレール①：正規表現のオートリペア（改行忘れ対策）
LLMは時折、出力フォーマットを間違えます。特に悪質なのが以下のような出力です。

```text
（LLMの実際の出力例）
...運営費や児童手当、育児休業給付、281
00:09:53,526 --> 00:09:55,962
あるいは障害児など
```

前のエントリのテキストの直後に、**改行を忘れて次のインデックス番号（281）をくっつけてしまっています。** これにより、通常のパーサーは「281\n00:09...」をすべて**前のテロップの文章の一部だと誤認**し、画面上にタイムコードがそのまま表示される大バグが発生します。

これを防ぐため、「数字と改行文字以外」の直後にSRTブロックが来た場合、強制的に改行を2つ挿入して修復する強力な正規表現フィルターをパーサーの直前に挟みました。

```python
# LLMが改行を忘れ、テキストの直後に「次のインデックス番号」を繋げた場合への対策
content = re.sub(
    r'([^\n\d])(\d+\s*\n\d{2}:\d{2}:\d{2}[,\.]\d{3}\s*-->)',
    r'\1\n\n\2',
    content
)
```
この `[^\n\d]` （改行でも数字でもない文字）という指定がキモです。これにより、日本語テキストの直後に来たタイムコードブロックだけをピンポイントで検知し、安全に分離することが可能になりました。

#### ガードレール②：タイムコード照合による「オフセット自動補正」
さらに恐ろしい事態があります。AIが「短すぎるから」という理由で勝手に字幕をいくつか削除し、**さらに番号を強制的に「1」から振り直して出力する**という二重の暴挙です。
位置情報（リストの上から何番目か）に頼ってタイムスタンプを復元しようとすると、この瞬間に後続のすべての字幕のタイムスタンプがズレ、動画全体が崩壊します。

AutoSRTでは、**AIが返してきたタイムスタンプの「HH:MM:SS」部分を解析し、原文のどの位置に該当するかを逆算**。ズレ幅（オフセット）を確定した上で、安全な「原文インデックス」ベースのタイムスタンプ復元を行います。

```python
# タイムスタンプ（HH:MM:SS）を利用して確実なオフセットを計算する
chunk_time_map = {e.start[:8]: e for e in chunk}
best_offset = None
for p in parsed:
    t = p.start[:8]
    if t in chunk_time_map:
        # LLMが1から振り直した番号と、原文の本当の番号の差分を確定
        best_offset = chunk_time_map[t].index - p.index
        break

if best_offset is not None and best_offset != 0:
    for entry in parsed:
        entry.index += best_offset  # LLMの番号を正しい番号に強制修正
```

万が一AIがテキストを破壊しすぎて照合不能になった場合（リストの長さも合わない場合）は、無理に適用せず**「その区間はAIの校正を破棄し、安全な原文をそのまま出力する」**というフォールバック設計にしています。これにより、**「絶対に映像と音をズラさない」というプロの現場の鉄則**を守り抜いています。

### 3. デスクトップ直接保存の実装

ブラウザ利用時の「ダウンロードフォルダに保存されたファイルを探すのが面倒」という地味なストレスは、Pythonの`pathlib`を使って解決しています。Webアプリとしてではなく、ローカル上で動かす社内ツールアプローチだからこそできる力技です。

```python
import pathlib
import streamlit as st

def _save_to_desktop(content: bytes, filename: str):
    desktop = pathlib.Path.home() / "Desktop"
    save_path = desktop / filename
    with open(save_path, "wb") as f:
        f.write(content)
    st.success(f"✅ デスクトップに保存しました: {filename}")
```
